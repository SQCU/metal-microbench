# Scalable oversight via bounded-depth recursive decomposition

**Status**: Design spec. Companion to
`docs/scalable_oversight_n_of_k.md` (the per-tool n-of-k summary
primitive). This doc covers the architectural pattern that USES the
n-of-k primitive: multi-stage validated workflows where a parent
agent's complete-and-validated outputs depend on sub-results
that NO SINGLE agent could produce inside its own context+token
budget while ALSO maintaining its conversational role.

## The setting

Concrete shape of the problem (paraphrased from the user's
2026-05-10 sketch):

> "make a complicated svg whose elements should be generated by
> passing 23 control points into a nontrivial python physics program
> to get a corresponding output svg which is finally rendered."

A model in scringlo's persona, with a 16k effective context for
serious task work, cannot:

1. Hold the conversational state + the persona prompt (~2k)
2. Write a non-trivial physics simulator in Python (~3-8k)
3. Choose 23 numerically-sensible control points (~1-2k of reasoning)
4. Validate the simulator against test cases (~2-4k of test output)
5. Render the final SVG (~2-5k)
6. Synthesize a response to the user that stays in her voice

…all inside one 16k window, and ALSO maintain conversational
character / not break persona / not produce gibberish in the seams
between sub-tasks.

But she CAN delegate. Each descendant uses ITS OWN 16k window for
ONE of those sub-tasks. The descendant is freed from the
conversational-persona constraint and can be a focused
"physics-simulator-writer" or "test-runner" or "control-point-chooser".
The aggregate work consumes 8×16k effective context — across
8 (or however many) workers — while the parent stays in scringlo
mode and sees only a bounded summary.

## Spawn vs. fork

Two flavors of descendant calls, with different semantics:

**Fork** (synchronous): parent's reply is BLOCKED on the descendant
result. Parent waits. The descendant's full result is available to
the parent once it returns. Used when the descendant's output
materially shapes the parent's next utterance — e.g., the parent
needs to KNOW whether the validator passed before saying "here's
your figure."

**Spawn** (asynchronous): parent fires the descendant and continues.
The descendant reports back later via the interrupt path
(see Phase 3c — `async-lookup` / `tool_async_result` injection).
Used when the descendant's work is supplementary — e.g., scringlo
fires off a "auto-improve-figure" descendant in the background, then
months later when it's done she shows the better version
("oh hey! i actually rendered a better one of those for you.").

In the recursive setting, fork-vs-spawn is a per-edge property of
the call graph. A typical workflow looks like:

```
Scringlo (parent)
  └─ FORK: physics-svg-compose
        ├─ FORK: design-physics-spec
        │     └─ FORK: validate-spec-numerically
        ├─ FORK: implement-python
        │     ├─ FORK: write-code
        │     └─ FORK: run-tests        ← could SPAWN if non-blocking
        │           ├─ test-1
        │           ├─ test-2
        │           ├─ test-3
        │           └─ test-N
        └─ FORK: render-svg
              └─ run-final-program
```

Bounded depth: 3-4 levels. Branching factor: 2-5 per level.
Total leaf calls: 8-30. Each leaf does ~1-3k tokens of focused work.
Aggregate ~50-100k tokens of work across leaves. Visible at scringlo
level: ~3 summary_trace lines (one per direct child).

## The interface question

Given:

- Parent agent (scringlo) has effective context budget B_p
- Each descendant subtree has total raw output L_subtree
- Parent receives n_subtree summary lines per subtree (default n=1)

Design constraints:

1. Sum of n_subtree across all direct children must fit in B_p.
   With B=16k and ~50 tokens per summary line, that's 320 lines max
   — but in practice we want ~3-10 so the parent has token headroom
   for its own response.

2. The user (human reader) gets the SAME lines the parent sees.
   This is the safety property: nothing the parent agent acts on is
   hidden from the user. (Exception: the descendant's full reasoning
   is available behind an expand affordance, but it's not in the
   parent's role:"tool" content.)

3. Drilling deeper costs extra prompt tokens. If the parent (or
   user) wants the grandchildren's summaries, that's an explicit
   request — UI: click-to-expand on the parent summary_trace; agent:
   "show me the implementation phase's sub-summaries" triggers a
   re-roll with deeper trace included.

### Concrete encoding

Each `summary_progress` event already carries `scope`. For
recursive composition, we extend `scope` to a path:

```
{
    "type": "summary_progress",
    "scope": "compose/design/validate-spec",
    "summary": "✅ verified spec is dimensionally consistent (3D positions, units checked)",
    "compressed_lines": 47,
    "depth": 3,
    "parent_session_id": "<the ancestor session this rolls up to>"
}
```

The FE driver bins by `scope` prefix when rendering — top-level
`compose/design` and `compose/implement` are visible at parent;
deeper-prefixed lines (e.g. `compose/design/validate-spec`) live in
nested expandables.

### Parent prompt-context discipline

When the parent reads the tool's role:"tool" result, it sees ONLY
the top-level summaries unless it explicitly asks for deeper. The
service composes the result.summary as:

```
[design]    ✅ designed pendulum sim, 23 control points chosen
[implement] ✅ python validates against 5 tests (MSE 0.001)
[render]    ✅ rendered 24-node svg, 8.2KB
[final]     here's your figure! the long-rope swing one ✨
```

That's 4 lines of role:"tool" content — bounded, accountable, parent-
voiced, faithful. The parent's own next response synthesizes from
those 4 lines plus the embedded final image.

If a stage failed, the corresponding line shows the failure plainly:

```
[design]    ✅ designed pendulum sim, 23 control points chosen
[implement] ❌ python failed 2/5 tests — energy conservation off by 8%
            ⤷ retried with corrected friction term, then ✅
[render]    ✅ rendered 24-node svg, 8.2KB
```

Honest accounting of the recursion's actual behavior, including
recoverable failures. Roll-up DOESN'T hide them.

## Why this is cheap (the cost model is the same as n-of-k)

Per the math in `scalable_oversight_n_of_k.md`:

- Each summary llm_call: ~50 tokens decode atop a shared-prefix
  prefill (the bridge's content-hash KV cache makes the prefill
  free for siblings sharing persona+history).
- Total summary cost across the tree: O(node_count × 50) decode
  tokens.
- Total descendant work cost: O(sum_of_leaf_decode_tokens), the
  thing we're paying for anyway because it's THE WORK.

Net oversight overhead: a constant multiplier on total decode work,
where the constant is small (~1-5%) because summaries are 50 tokens
each and leaves do 1-3k.

The bridge's existing parallel-streams capability (we measured
~2.64x speedup on 3 concurrent streams) means sibling descendants
overlap upstream — so wall-clock time is dominated by the tree's
critical path, not the sum of all node times.

## Implementation plan (this design's first deliverable)

A single card, `physics-svg-compose`, demonstrating:

1. **3-level bounded recursion**:
   - Level 0: scringlo (parent agent) calls the card
   - Level 1: card decomposes into 3 stages (design / implement / render)
     — these 3 are FORKED in sequence (the implement depends on
     design's output, render depends on implement's)
   - Level 2: implement stage forks 3-5 test-runner sub-calls in
     parallel via parallel_llm_call

2. **Validation at boundaries**: each stage's output must
   pass a verification step before being passed to the next. Failures
   surface as 💭 lines in the parent's trace + the parent agent's
   own next response can mention them.

3. **n_visible bounded**: parent sees 3-4 trace lines for a workflow
   producing 100s of lines of intermediate work.

4. **Real-pixel video**: test 17 drives Scringlo through the full
   recursion, captures video showing the trace landing depth-first
   as descendants complete.

### Card shape

`physics-svg-compose` manifest:
- one tool: `compose(description: string)` →
  returns `{summary, image_url, design_spec, validation_report}`

### Service stages

Each stage is a fork (synchronous descendant llm_call sequence):

**Stage A: design (`compose/design`)**
- LLM call 1: "given description, propose a Python physics function
  signature, identify control points, list dimensional/numerical
  invariants the program must satisfy"
- LLM call 2: "validate the spec by reviewing the invariants for
  consistency"
- emit `summary_progress(scope='compose/design', ...)` with one-line
  parent-voice summary

**Stage B: implement (`compose/implement`)**
- LLM call 1: "write the Python function matching the design spec"
- emits intermediate `summary_progress(scope='compose/implement/write')`
- Sub-fork: parallel test runners (level-2 recursion)
  - For each of N=5 test cases: a `test-runner` sub-call evaluates
    the function with crafted inputs and validates outputs
  - Each test runner emits its own per-test summary_progress
- emit roll-up `summary_progress(scope='compose/implement', ...)` aggregating

**Stage C: render (`compose/render`)**
- Run the validated Python with the chosen control points (host-side,
  NOT another LLM call — this is the pure execution step)
- Emit `summary_progress(scope='compose/render', ...)` with results

Final result.summary concatenates:
```
[design]      <line>
[implement]   <line + test-pass count>
[render]      <line + svg size>
[final]       <one-line scringlo-voice wrap-up>
```

### Test 17: real-pixel demo

Drive the test-16 setup with a complex physics request:
"make me an svg of a damped harmonic oscillator with 5 nodes"

Expect:
- Total wall: 60-90s (3 stages, each with 1-3 LLM calls + parallel
  tests)
- Parent's summary_trace: ~4 lines
- Tool result: SVG image rendered inline in scringlo's bubble
- Frame extraction: confirm summary_trace lines land progressively
  (one per stage), final image embedded

## What this proves vs. doesn't

**Proves**:
- Bounded-depth recursive decomposition is buildable on the
  existing toolcards substrate.
- Each level's n-of-k summary chain composes correctly: parent's
  trace is bounded regardless of total leaf work.
- 8x16k aggregate context can be assembled by gemma4 instances
  that individually can't hold all of it. Each leaf's own context
  budget is respected.

**Doesn't prove**:
- That the resulting SVG is artistically good. Quality is a
  separate axis from this scaffolding.
- That the validation steps catch all bugs. They catch the
  invariants we encode; latent bugs (numerical instability across
  parameter regions, etc.) require richer testing.
- That the cost remains tractable for trees of arbitrary breadth.
  We measure the small case (depth=3, branch=5) and extrapolate;
  larger trees would need explicit batching budget management.

**Open architectural questions**:
- Should the card surface a "drill into level 2 summaries" affordance
  natively, or is that a separate FE feature (UI: click-to-expand)?
  Latter is simpler; can ship first.
- Forced re-validation on retry: if an intermediate stage fails and
  retries, do we surface BOTH the failure line and the recovery
  line, or just the recovery? Honest answer: BOTH. Hiding the
  failure undermines the oversight property.
- Cross-tree information flow: when stage B's tests reveal a flaw
  in stage A's spec, does the card auto-rollback to A and re-design,
  or does it surface the failure to the parent? Auto-rollback at
  one level of depth is fine; deeper rollback is a separate
  feature (graph rewriting). Defer.
